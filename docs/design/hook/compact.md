# 压缩机制

参考： /usr1/DataCom-SDD/ref/oh-my-openagent 其中的压缩实现方法（实现主动压缩）

我这里做了总结文档

```markdown
根据代码分析，这个项目的压缩机制是一个预防性会话精简系统，主要实现如下：

  核心流程

  1. 触发检测 (preemptive-compaction-trigger.ts)

  在 tool.execute.after hook 中，每次工具执行后检查是否需要压缩：

  - 阈值：当 token 使用率达到 78% (PREEMPTIVE_COMPACTION_THRESHOLD = 0.78) 时触发
  - 冷却期：同一会话两次压缩间隔至少 60 秒
  - 超时保护：压缩操作有 60 秒超时限制

  const usageRatio = totalInputTokens / actualLimit
  if (usageRatio < PREEMPTIVE_COMPACTION_THRESHOLD) return

  2. 压缩执行

  调用 ctx.client.session.summarize() API 进行实际的会话精简，使用专门的压缩模型（可配置）。

  ---
  三个关键组件
  
  Context Preserver (compaction-context-injector)

  负责在压缩前后捕获和恢复会话状态：

  1. Capture (压缩前)：保存当前的 agent、model、tools 配置
  2. Inject：压缩时将背景任务的上下文注入到新会话
  3. Restore (压缩后)：恢复之前保存的 agent 配置

  还包括监控机制：跟踪助手回复的文本文本尾部，如果压缩后连续多条消息没有文本输出，会发出警告。

  Todo Preserver (compaction-todo-preserver)

  专门保护任务列表 (TodoList) 不被丢失：

  - 压缩前快照当前 todos
  - 压缩后自动恢复，且智能判断是否需要覆盖当前列表
  - 特别处理 Atlas 的 bootstrap todos，防止被空的默认列表覆盖
  - 在 tool.execute.before hook 中拦截 todowrite 调用，确保恢复后的快照不被意外覆盖

  退化监控 (degradation-monitor)

  监控压缩后是否出现模型退化（例如连续生成空内容）：

  - 压缩后监听 5 条助手消息
  - 如果连续 3 条消息检测到 "no-text tail"，触发恢复性压缩
  - 最多重试 3 次
  - 通过 UI toast 通知用户正在尝试恢复

  ---
  自动继续处理 (session-compacting.ts)
  
  处理 experimental.compaction.autocontinue hook：
  - 非压缩 agent 才允许自动继续
  - 调用 context injector 和 todo preserver 的 restore 方法
  - 实现无感知的会话状态恢复

  ---
  架构亮点
  
  ┌────────────────┬───────────────────────────────────────────────┐
  │      特性      │                     实现                      │
  ├────────────────┼────────────────────────────────────────────────────┤
  │ 上下文限制识别 │ 动态解析不同 provider/model 的实际 token 上限      │
  ├────────────────┼────────────────────────────────────────────────────┤
  │ 降级保护       │ 压缩失败时显示 toast 警告，不影响主流程            │
  ├────────────────┼────────────────────────────────────────────────────┤
  │ 防止递归       │ 识别压缩 agent (isCompactionAgent)，跳过对其的压缩 │
  ├────────────────┼────────────────────────────────────────────────────┤
  │ 会话清理       │ session.deleted 事件时清理所有缓存状态             │
  └────────────────┴────────────────────────────────────────────────────┘

  整个设计保证了长会话能够自动、无损地维护上下文，避免因 token 超限导致的会话中断。
```
